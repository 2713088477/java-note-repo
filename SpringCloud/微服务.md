# 微服务

## 1.认识微服务

微服务是一种软件架构风格，它是以专注于单一职责的很多小型项目为基础，组合出复杂的大型应用。

可能会设计到以下问题:

- 服务拆分
- 远程调用
- 服务治理
- 请求路由
- 身份认证
- 配置管理
- 服务保护
- 分布式事务
- 异步通信
- 消息可靠性
- 延迟消息
- 分布式搜索
- 倒排索引
- 数据聚合

### 1.1单体架构

**单体架构**：将业务的所有功能都集中在一个项目中开发，打成一个包部署。

![](assets\单体架构图片.png)

优点：

1) 架构简单
2) 部署成本低

缺点:

1) 团队协作成本高
2) 系统发布效率差
3) 系统可用性差，一个jvm并发量有限，一个接口的高并发影响其他接口

总结:单体架构适合开发功能相对简单，规模较小的项目。

### 1.2微服务架构



微服务架构，首先是服务化，就是将单体架构中的功能模块从单体应用中拆分出来，独立部署为多个服务。同时要满足下面的一些特点:

- **单一职责**:一个微服务负责一部分的业务功能，并且其核心数据不依赖于其他模块。
- **团队自治**:每个微服务都有独立的开发、测试、发布、运维人员，团队人员不超过10人。
- **服务自治**：每个微服务都独立打包部署，访问自己独立的数据库。并且要做好服务隔离，避免对其他服务产生影响。

## 2.微服务拆分的原则

拆分目标:

1. **高内聚**:每个微服务的职责要尽量单一，包含的业务相互关联度高、完整度高
2. **低耦合:**每个微服务的功能要相对独立，尽量减少对其他微服务的依赖

从拆分方式来说，一般包含两种方式:

1. **纵向拆分:** 按照业务模块来拆分
2. **横向拆分:** 抽取公共服务，提高复用性

工程结构:

1. 独立Project
2. Maven聚合 

### 2.1拆分过程中的远程调用

通过使用`RestTemplate`来发送网络请求

```java
@MapperScan("com.hmall.cart.mapper")
@SpringBootApplication
public class CartApplication {
    public static void main(String[] args) {
        SpringApplication.run(CartApplication.class,args);
    }


    @Bean
    public RestTemplate restTemplate(){
        return new RestTemplate();
    }
}

```

示例代码:

```java
private void handleCartItems(List<CartVO> vos) {

    // 1.获取商品id
    Set<Long> itemIds = vos.stream().map(CartVO::getItemId).collect(Collectors.toSet());
    // 2.查询商品,发送Http请求
    ResponseEntity<List<ItemDTO>> response = restTemplate.exchange(
            "http://localhost:8081/items?ids={ids}",
            HttpMethod.GET,
            null,
            new ParameterizedTypeReference<List<ItemDTO>>() {
            },
            Map.of("ids", CollUtils.join(itemIds, ","))
    );
    //查询失败直接返回
    if (!response.getStatusCode().is2xxSuccessful()) {
        return;
    }
    List<ItemDTO> items = response.getBody();
    // 3.转为 id 到 item的map
    Map<Long, ItemDTO> itemMap = items.stream().collect(Collectors.toMap(ItemDTO::getId, Function.identity()));
    // 4.写入vo
    for (CartVO v : vos) {
        ItemDTO item = itemMap.get(v.getItemId());
        if (item == null) {
            continue;
        }
        v.setNewPrice(item.getPrice());
        v.setStatus(item.getStatus());
        v.setStock(item.getStock());
    }
}
```

## 2.3注册中心原理

![](assets\注册中心的原理.png)



服务治理中的三个角色:

- **服务提供者**：暴露服务接口，供其它服务调用  
- **服务消费者**：调用其它服务提供的接口  
- **注册中心**：记录并监控微服务各实例状态，推送服务变更信息

消费者如何知道提供者的地址？

- 服务提供者会在启动时注册自己信息到注册中心，消费者可以从注册中心订阅和拉取服务信息 

消费者如何得知服务状态变更？

- 服务提供者通过心跳机制向注册中心报告自己的健康状态；当心跳异常时，注册中心会将异常服务剔除，并通知订阅了该服务的消费者

当提供者有多个实例时，消费者该选择哪一个？

- 消费者可以通过**负载均衡算法**，从多个实例中选择一个（如轮询、随机、加权最小连接数、一致性哈希等）
